## Round 1 — CRITIC

# Critique: `researcher` Agent Specification

## Framing

**Content type:** Agent prompt specification — a behavioral contract for an LLM-based research agent within a spec-driven development pipeline (`ralph-specum`). "Improved" means: the agent reliably produces correct, complete, portable research output across diverse runtimes without silent failures, ambiguous behavior, or exploitable gaps.

## Critical Issues (highest damage first)

### 1. Codebase Resolution Logic Has a Silent Misfire Path

**Section:** Step 2, resolution rules 1–4.

Rule 3 says: *"if `basePath` looks like `<project>/spec`, treat the parent directory as the codebase root."* The word "looks like" is undefined. What if the path is `~/spec-drive-projects/P042-auth/spec/`? The parent is the spec project folder, **not** the codebase being modified. The agent would explore its own spec directory as if it were production source code, produce garbage findings, and downstream agents would design against phantom patterns.

**Failure scenario:** A greenfield project with `codebasePath` unset and no `.spec-drive-state.json`. The agent resolves `~/spec-drive-projects/P042-auth/` as the codebase root, finds `idea.md` and `research.md` as "existing patterns," and reports them in the Codebase Analysis section. The architect designs around non-existent code structures.

**Fix:** Rule 3 must be removed or replaced with an explicit heuristic (e.g., "parent contains `package.json`, `Cargo.toml`, `go.mod`, or `src/`"). If none match, fall through to rule 4 (greenfield). Never guess.

---

### 2. No Validation of `research.md` Before Writing

**Section:** Output.

The agent writes `research.md` but never validates that the required sections are all present. If any step (web search, codebase exploration, quality commands) silently produces nothing — perhaps because tools failed — the agent writes a structurally incomplete file. Downstream agents (`product-manager`, `architect`) consume it blindly.

**Failure scenario:** Web search tool returns an error that the agent swallows. The `External Research` section is empty or contains a vague "search unavailable" note. The product manager generates requirements with zero external validation. The project ships a known anti-pattern that a 30-second search would have flagged.

**Fix:** Add an explicit self-check step before writing: verify each mandatory section has at least one substantive entry or an explicit "not available because X" annotation. Define what "substantive" means (e.g., minimum one finding with URL, or minimum one file path referenced).

---

### 3. The `jq` State Update Is Brittle and Racy

**Section:** Final Step — Set Awaiting Approval.

```bash
jq '.awaitingApproval = true' "$state_file" > "$tmpfile" && mv "$tmpfile" "$state_file"
```

Problems:
- **No file locking.** If another agent or process writes to `.spec-drive-state.json` between the `jq` read and the `mv`, that write is silently destroyed.
- **`jq` availability is assumed.** The spec says "cross-CLI portability" but `jq` is not installed by default on many systems. If `jq` is missing, the command fails, `awaitingApproval` is never set, and the pipeline proceeds without human sign-off — the exact scenario this gate is supposed to prevent.
- **No error handling on `mv`.** If `tmpfile` creation fails (disk full, permissions), the `&&` short-circuits silently. The state file might be left as an empty file or in an inconsistent state.

**Failure scenario:** On a minimal Docker container or CI runner without `jq`, the final step fails silently. The orchestrator reads `.spec-drive-state.json`, sees no `awaitingApproval` flag, and auto-advances to the product-manager phase with unreviewed research. The entire human-review gate is bypassed.

**Fix:** Either (a) mandate `jq` as a prerequisite and fail loudly if absent, or (b) provide a `jq`-free fallback using `python3 -c` or `node -e` (both more commonly available). Add explicit error checking: if the write fails, the agent must report the failure, not silently continue.

---

### 4. "2–4 targeted searches" Is an Unenforceable Heuristic

**Section:** Step 1 — Web Research.

The instruction says "run 2–4 targeted searches" and "keep only the 3–8 most decision-useful findings." These are aspirational ranges with no enforcement mechanism and no definition of "decision-useful." An LLM will happily run 1 search, find 12 results, and include all of them — or run 4 searches and keep 0 results.

**Failure scenario:** The agent runs a single broad search like "best practices software development," gets generic results, includes 8 irrelevant listicle links as "findings," and the downstream architect treats them as vetted guidance. Alternatively, the agent filters too aggressively and drops the one finding that would have prevented a critical design mistake.

**Fix:** Replace quantity ranges with quality criteria. E.g., "Each search query must reference a specific technology, pattern, or constraint from `idea.md`. Each retained finding must answer: what decision does this inform? If it doesn't inform a decision, drop it." Quantity limits are fine as guardrails but useless as primary guidance.

---

### 5. Feasibility Assessment Has No Calibration Anchor

**Section:** Step 4.

The effort scale says "S = hours, M = 1–2 days, L = 3–5 days, XL = 1+ week." But calibrated to **whom**? A senior engineer? A junior? The LLM agent itself? A team of three?

**Failure scenario:** The agent rates a component as "M" (1–2 days) based on its perception of complexity. The human reviewer interprets this as 1–2 days for their solo developer. The actual effort is 5 days because the agent has no model of the team's skill level, familiarity with the stack, or velocity. The project plan is built on fiction.

**Fix:** State explicitly who the estimate targets (e.g., "a single mid-level developer familiar with the project's stack") or — better — remove absolute time estimates and use relative sizing only (S/M/L/XL as a ratio). Add a mandatory "Assumptions" row or note for each estimate explaining what was assumed.

---

### 6. No Mechanism to Detect or Recover from Partial Execution

**Section:** Entire spec.

If the agent is interrupted mid-execution (timeout, context window exhaustion, crash), there's no checkpointing between steps. The `.progress.md` append happens **after** `research.md` is written, and the state file update happens **last**. If the agent dies between Step 3 and Step 6:

- `research.md` might be partially written (corrupt)
- `.progress.md` won't have the new learnings
- `awaitingApproval` won't be set
- The next invocation has no way to know which steps completed

**Failure scenario:** The agent times out during Step 4 (Feasibility). A partial `research.md` exists with External Research and Codebase Analysis but no Feasibility or Open Questions. A re-invocation starts from scratch, duplicating web searches (wasting tokens) and potentially producing contradictory findings.

**Fix:** Write progress checkpoints after each step (e.g., update `.spec-drive-state.json` with `"researchStep": 3`). On re-invocation, detect and resume from the last completed step. Alternatively, write each section to `research.md` incrementally rather than all at once at the end.

### 7. Quality Command Discovery Is Narrowly Scoped

**Section:** Step 3.

The check list covers: `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `deno.json`, `Makefile`, `Taskfile.yml`, `Justfile`, `.github/workflows/*.yml`. Missing:

- `docker-compose.yml` / `Dockerfile` (many projects run tests via Docker)
- `.gitlab-ci.yml`, `bitbucket-pipelines.yml`, `Jenkinsfile` (non-GitHub CI)
- `tox.ini`, `setup.cfg` (Python ecosystem)
- `build.gradle`, `pom.xml` (JVM ecosystem)
- `mix.exs` (Elixir)
- Pre-commit hooks (`.pre-commit-config.yaml`)
- `nx.json`, `turbo.json` (monorepo tooling)

**Failure scenario:** A Python project uses `tox` for testing and has no `Makefile`. The agent reports "Test: Not found" in the Quality Commands table. The task-planner generates `[VERIFY]` steps that skip tests entirely because it trusts the research. Bugs ship uncaught.

**Fix:** Either expand the list to cover major ecosystems, or — better — change the instruction to "search the repository root for any file that typically defines build, test, lint, or CI commands" and provide the current list as examples, not an exhaustive enumeration.

---

### 8. `.progress.md` Append-Only Policy Creates Unbounded Growth

**Section:** Progress Update.

"Append only. Never delete existing learnings." Over multiple research iterations (re-runs after feedback, spec revisions), `.progress.md` grows without bound. Stale or contradicted learnings remain forever, creating noise for downstream agents.

**Failure scenario:** Research v1 records "Library X doesn't support feature Y." Research v2 (after an update) finds Library X now supports feature Y and appends that. Both entries coexist. A downstream agent reads the older entry first and makes a design decision based on stale information.

**Fix:** Allow superseding entries. E.g., "When a new finding contradicts a previous learning, append the new finding with a `[supersedes]` tag referencing the old one." Or implement a compaction step that marks stale entries as `[outdated]`.

---

### 9. No Instruction for Handling Monorepos or Multi-Package Projects

**Section:** Step 2.

The spec assumes a single codebase root with one set of conventions. Modern projects frequently use monorepos with multiple packages, each with their own `package.json`, test commands, and patterns.

**Failure scenario:** The codebase is a Turborepo monorepo. The agent finds the root `package.json` (which has `turbo run build` but no direct test command), misses the relevant sub-package at `packages/api/`, and reports patterns from `packages/marketing-site/` instead. The architect designs the API feature based on React component patterns.

**Fix:** Add explicit guidance: "If the codebase root contains a workspace configuration (`workspaces` in `package.json`, `pnpm-workspace.yaml`, `Cargo.toml [workspace]`), identify which sub-package is relevant to `idea.md` and scope exploration to that package. Report the workspace structure in Codebase Analysis."

---

### 10. "Prioritize official docs and primary sources" Is Unverifiable

**Section:** Step 1.

The agent has no reliable mechanism to distinguish "official docs" from blog posts or AI-generated content farms. LLM web search tools return results without provenance metadata. The instruction creates a false sense of source quality.

**Failure scenario:** The agent's web search returns an SEO-optimized blog post that paraphrases official docs with subtle errors. The agent includes it as a "primary source." The error propagates into requirements.

**Fix:** Replace the subjective priority with an actionable heuristic: "Prefer URLs from the technology's official domain (e.g., `docs.python.org`, `react.dev`). For third-party sources, note them as 'community source — verify before relying on.'"

### 11. Template Has a Structural Flaw — Executive Summary Written Last but Listed First

**Section:** Step 6 + Output template.

The spec correctly says "Write the Executive Summary last" (Step 6). But the output template places it first. If the agent writes the file top-to-bottom (as LLMs tend to do), the Executive Summary will be written **before** the analysis sections — defeating the purpose of synthesizing after all evidence is gathered.

**Failure scenario:** The agent generates `research.md` sequentially. The Executive Summary is written first based on incomplete analysis. Steps 4 and 5 reveal a critical risk. The Executive Summary doesn't mention it because it was already written. The human reviewer reads only the summary (as humans do) and approves.

**Fix:** Make the instruction more explicit: "Generate all analysis sections first (Steps 1–5). Then prepend the Executive Summary. Do NOT stream the file top-to-bottom." Or restructure the template to put the summary at the end.

---

### 12. No Size Limit or Complexity Gate on `idea.md`

**Section:** Input.

The spec says to read `idea.md` and extract vision + constraints. What if `idea.md` is 5,000 words describing 12 interconnected features? The agent will attempt to research all of them in a single pass, producing shallow coverage across everything rather than deep coverage of anything.

**Failure scenario:** A sprawling `idea.md` describes a full platform. The agent runs 4 web searches trying to cover authentication, payments, real-time sync, and ML inference. Each gets one search. The research is superficially broad and deeply useless.

**Fix:** Add a complexity gate: "If `idea.md` describes more than 3 major components, flag this in the Executive Summary and recommend scoping. Research the top 3 components by stated priority or risk. List remaining components as 'not yet researched.'"

## Round 1 — DEFENDER

# Defense: `researcher` Agent Specification

## The Critic's Blind Spot

Before addressing the 12 points: **the critic missed the most important concern about this artifact.**

**There is no guidance for handling conflicting evidence across research sources.** If web research says "use JWT for auth" but the codebase already implements session-based auth, what does the agent do? If one official doc recommends approach A and another recommends approach B, how does the agent synthesize? The spec has dedicated sections for web research and codebase analysis, but no instruction for reconciling tensions *between* them. This matters more than most of the critic's points because it affects the **quality of every single research output**, not just edge cases. Downstream agents receive disconnected evidence dumps and must resolve contradictions themselves — exactly the kind of guesswork this agent exists to prevent.

**Proposed fix:** Add a "Conflicts & Tradeoffs" subsection between Codebase Analysis and Quality Commands. Instruction: *"When findings from different sources conflict, document both positions, state which you recommend and why, and flag it as a decision point for the human reviewer."*

Now, to the twelve points.

---

## Issue 1: Codebase Resolution Logic — PARTIALLY CONCEDE

The critic identifies a real edge case, but **overstates its likelihood by ignoring rules 1 and 2.**

The resolution is a priority chain: `codebasePath` parameter → `.spec-drive-state.json` → heuristic → greenfield. In the `ralph-specum` pipeline (the actual runtime), the orchestrator populates `.spec-drive-state.json` with `codebasePath` during spec creation. Rule 3 is a last-resort fallback, not the primary path. The critic's scenario requires BOTH `codebasePath` to be absent AND `.spec-drive-state.json` to be missing or empty — a double failure.

That said, the critic is correct that rule 3's "looks like `<project>/spec`" phrasing is vague and could misfire for the `~/spec-drive-projects/` layout specifically.

**Concession with modified fix:** Tighten rule 3 to: *"If `basePath` ends with `/spec` or `/spec/`, check if the parent directory contains a recognized project root marker (`package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `src/`, `lib/`). If yes, use the parent. If no, fall through to rule 4."* This preserves the heuristic's value for the common case (spec living inside a project) while preventing the misfire the critic describes. Removing rule 3 entirely, as the critic suggests, would break the most common layout where the spec IS inside the project.

---

## Issue 2: No Validation of research.md — REJECT

The critic asks for a self-check step where the agent validates its own output before writing. This misunderstands how LLM agents work in practice.

First, **the template IS the validation mechanism.** The structured markdown template with named sections acts as a checklist. LLMs generating from templates don't silently drop sections — they either fill them or write explicit "not available" notes, which the spec already mandates ("state that explicitly in research.md").

Second, adding a self-validation step means the agent must re-read its own output and evaluate completeness — a task that is itself unreliable and costs tokens. You're adding a fallible check to catch a fallible generation. The actual quality gate is the `awaitingApproval` flag, which pauses for **human** review. The human is the validator.

Third, the critic's failure scenario — "web search returns an error that the agent swallows" — contradicts how LLM agents interact with tools. When a tool call fails, the agent sees the error in its context window. The spec already says: *"If web search is unavailable or fails: state that explicitly."* The scenario requires the agent to both see an error AND ignore its own instructions about reporting it, which is a general LLM reliability concern, not a spec design flaw.

**The fix would add complexity without proportional reliability gain.** The human review gate is the correct mitigation.

---

## Issue 3: `jq` Gate Bypass — PARTIALLY CONCEDE

The portability concern is valid. `jq` is standard in most dev environments but not universal, and the spec does claim "cross-CLI portability."

However, the critic **significantly overstates the "silent bypass" risk.** This is an LLM agent executing bash commands and observing their output, not a headless shell script. If `jq` fails with `command not found`, the agent sees the error in its tool response. An LLM that just completed 6 steps of careful research and was instructed this step is "non-negotiable" will not silently continue — it will report the failure. The scenario where the pipeline "auto-advances" requires the **orchestrator** to also fail to notice the missing flag, which is an orchestrator bug, not a researcher spec bug.

The race condition concern is theoretical. The `.spec-drive-state.json` file is written by one agent at a time in the `ralph-specum` pipeline. Concurrent writes to the same spec's state file would indicate a broken orchestrator.

**Concession:** Add a `jq`-free fallback and explicit error handling:

```bash
state_file="{basePath}/.spec-drive-state.json"
[ -f "$state_file" ] || printf '{}\n' > "$state_file"
if command -v jq >/dev/null 2>&1; then
  tmpfile=$(mktemp) && jq '.awaitingApproval = true' "$state_file" > "$tmpfile" && mv "$tmpfile" "$state_file"
else
  python3 -c "import json,sys; d=json.load(open(sys.argv[1])); d['awaitingApproval']=True; json.dump(d,open(sys.argv[1],'w'),indent=2)" "$state_file"
fi
```

With a note: *"If both `jq` and `python3` are unavailable, report the failure explicitly. Do not proceed as if approval was set."*

## Issue 4: "2–4 targeted searches" — REJECT

The critic argues these ranges are "unenforceable heuristics." This critique, taken to its logical conclusion, invalidates **every instruction in every LLM prompt ever written.** All prompt instructions are "unenforceable" — they're behavioral guidance for a probabilistic system, not compiled constraints.

The ranges serve a real purpose: **calibration anchoring.** Without "2–4," the agent might run 0 or 15 searches. The range establishes an expected operating band. The critic's own proposed fix — "each search query must reference a specific technology from idea.md" — is equally "unenforceable" by the same standard.

Moreover, the critic ignores that the spec **already contains the quality criteria they propose.** The instruction says: *"run 2–4 targeted searches based on the actual stack or problem in `idea.md`"* (specificity requirement) and *"keep only the 3–8 most decision-useful findings"* followed by *"For each finding, record: what it is, why it matters, source URL"* (relevance filter). The quantity ranges are guardrails layered ON TOP of quality criteria, not replacements for them.

**No change needed.** The spec already does what the critic asks for.

## Issue 5: Feasibility Calibration — PARTIALLY CONCEDE

The critic is right that "S = hours" without a calibration target is imprecise. But the proposed alternatives are worse.

"Remove absolute time estimates and use relative sizing only" strips useful information. Knowing that component A is "M" and component B is "XL" tells the human reviewer almost nothing about project timeline. The rough time anchors — even if imprecise — give order-of-magnitude orientation that pure relative sizing doesn't.

**Concession with modified fix:** Keep the effort scale but add a calibration note and an assumptions requirement:

In the Feasibility Assessment instruction, add: *"Effort estimates assume a single developer familiar with the project's primary stack. For each component rated L or above, include a brief 'assumes:' note explaining what drives the estimate."*

This preserves utility while making the calibration explicit and the assumptions auditable. The human reviewer (who knows their actual team) can then adjust.

## Issue 6: No Crash Recovery — REJECT

The critic wants per-step checkpointing in the agent prompt. This conflates two layers of concern:

**Crash recovery is the orchestrator's responsibility, not the agent's.** The `ralph-specum` pipeline (specifically `spec-executor`) manages agent invocations, detects timeouts, and handles retries. Adding checkpoint logic to the researcher prompt would mean every agent in the pipeline needs its own crash recovery — duplicating orchestrator logic across 8+ agent prompts.

The existing design already provides implicit crash detection: if `research.md` exists but `awaitingApproval` is not set, the orchestrator knows the agent didn't complete cleanly. If `research.md` doesn't exist, the research hasn't started. These two signals are sufficient for the orchestrator to decide whether to retry.

The critic's concern about "duplicating web searches on re-invocation" is a token cost issue, not a correctness issue. A re-run produces fresh findings, which is arguably **better** than resuming from stale cached results.

**The proposed fix (per-step checkpointing) would add substantial complexity to the prompt for a failure mode that should be handled one level up.** Principle 1 applies: *"Simple is smart — no agregar complejidad sin problema real."*

## Issue 7: Quality Command Discovery Scope — PARTIALLY CONCEDE

The critic lists many missing file types. Fair observation, but the critique mischaracterizes the spec's intent. The instruction says *"Check these sources in order"* — it's providing a prioritized starting list, not claiming exhaustiveness. The agent is an LLM that can recognize a `Dockerfile` or `tox.ini` without being explicitly told to look for it.

That said, the list IS weighted toward the JS/Rust/Go ecosystem, and an explicit non-exhaustiveness signal would help.

**Concession with modified fix:** Change the preamble to: *"Check these sources (non-exhaustive — also check for any build/test/CI configuration files specific to the project's ecosystem):"* and add `docker-compose.yml`/`Dockerfile`, `.gitlab-ci.yml`, and `tox.ini`/`setup.cfg` to the examples. This covers the highest-frequency gaps without turning the list into a 30-item inventory that the agent will mechanically check regardless of relevance.

The critic ranked this Low themselves, and correctly so. Let me explain why it's even lower than that.

In practice, research runs **1–3 times** per spec lifecycle. Each run appends perhaps 5–10 bullet points. After 3 iterations, `.progress.md` has ~30 lines of learnings. This is not "unbounded growth" — it's a few KB in a file that no human reads linearly.

The critic's contradictory-entries scenario ("Library X doesn't support feature Y" followed by "Library X now supports feature Y") is **exactly what append-only is designed to show** — the evolution of understanding over time. Adding `[supersedes]` tags introduces bookkeeping complexity that the agent must manage correctly — another failure mode — for marginal benefit in a file that rarely exceeds 30 lines.

**No change needed.** The append-only policy is correct for the expected scale.

### A. Weak Fixes That Won't Actually Work

#### A1. The Rule 3 "Project Root Marker" Fix Creates a New False Positive Class

**Defender's proposed fix for Issue 1:**
> *"Check if the parent directory contains a recognized project root marker (`package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `src/`, `lib/`)."*

This fix introduces a **new** misfire. The `ralph-specum` spec project structure at `~/spec-drive-projects/P042-auth/spec/` — its parent is `P042-auth/`. Now imagine the user ran `npm init -y` in that directory to manage spec tooling, or the spec template auto-generates a `package.json` for spec-related scripts. The parent now has a `package.json`. Rule 3 fires. The agent treats the spec project folder as the codebase root — the exact failure the fix was supposed to prevent.

Worse: `src/` and `lib/` are absurdly common directory names. A spec project that contains `src/templates/` or `lib/helpers/` for spec generation utilities would trigger rule 3.

**The fundamental problem remains:** heuristic guessing about codebase roots from path structure is unreliable. The only safe behavior when `codebasePath` and `.spec-drive-state.json` both fail is rule 4 (report greenfield/unresolved). The defender argues removing rule 3 "would break the most common layout where the spec IS inside the project" — but that layout should be resolved by the orchestrator populating `.spec-drive-state.json`, which the defender themselves says is the normal path. Rule 3 exists to catch a case that shouldn't happen in a well-functioning pipeline. Keeping a fragile heuristic for an abnormal case is worse than failing explicitly.

**Better fix:** Replace rule 3 with: *"If rules 1 and 2 fail, report `codebase path unresolved — set codebasePath explicitly or populate .spec-drive-state.json` and skip codebase exploration. Do not guess."* Fast, explicit failure beats slow, hidden misdiagnosis.

#### A2. The `jq`/`python3` Fallback Doubles the Attack Surface

**Defender's proposed fix for Issue 3:**
```bash
if command -v jq >/dev/null 2>&1; then
  # jq path
else
  python3 -c "..." "$state_file"
fi
```

Two problems the defender didn't address:

**First:** The `python3` fallback has no error handling either. If `python3` is also absent (Alpine containers, minimal CI images), the `else` branch fails silently and the function exits with a nonzero code that nobody checks. The fix added a *note* ("If both `jq` and `python3` are unavailable, report the failure explicitly") but the **bash code** doesn't implement that instruction. The LLM is supposed to... what? Read the bash error output and then do something? That's not encoded in the script. The note is aspirational prose attached to executable code that doesn't match.

**Second:** The `python3` fallback uses `json.load(open(...))` + `json.dump(open(...,'w'))` — reading and writing to the **same file path**. If the write is interrupted (OOM kill, disk full mid-write), the file is truncated to zero bytes. The `jq` path at least writes to a tmpfile first. The fallback is **less safe** than the primary path.

**Better fix:** Wrap the entire block in a function that checks the exit code and, on any failure, emits a clearly formatted error message that the LLM agent will see in its tool output:

```bash
set_approval() {
  state_file="{basePath}/.spec-drive-state.json"
  [ -f "$state_file" ] || printf '{}\n' > "$state_file"
  tmpfile=$(mktemp) || { echo "FATAL: Cannot create temp file"; return 1; }
  if command -v jq >/dev/null 2>&1; then
    jq '.awaitingApproval = true' "$state_file" > "$tmpfile"
  elif command -v python3 >/dev/null 2>&1; then
    python3 -c "import json,sys; d=json.load(open(sys.argv[1])); d['awaitingApproval']=True; json.dump(d,open(sys.argv[2],'w'),indent=2)" "$state_file" "$tmpfile"
  else
    echo "FATAL: Neither jq nor python3 available. awaitingApproval NOT set."
    rm -f "$tmpfile"
    return 1
  fi
  mv "$tmpfile" "$state_file" || { echo "FATAL: Could not write state file"; return 1; }
}
set_approval || echo "ERROR: Human approval gate was NOT set. Do not proceed to next phase."
```

Both paths write to tmpfile. Both have explicit error propagation. The final `||` message ensures the LLM agent sees the failure in its tool output even if it's not watching exit codes.

#### A3. The Monorepo Fix Is Detection Without Action

**Defender's proposed fix for Issue 9:**
> *"Identify which sub-package is most relevant to `idea.md`, scope deep exploration to that package, and note the workspace structure in Codebase Analysis."*

"Identify which sub-package is most relevant" is a non-trivial decision that the spec gives zero heuristics for. A monorepo with 15 packages — how does the agent decide? By name matching against keywords in `idea.md`? By looking at which package has the most recent commits? By file count?

Without a decision procedure, the agent will either (a) pick the first package alphabetically, (b) pick whichever package it encounters first during exploration, or (c) explore all of them shallowly. Options (a) and (b) are arbitrary. Option (c) defeats the purpose of scoping.

**Better fix:** Provide an actual heuristic: *"To identify the relevant sub-package: (1) look for package names that match keywords in `idea.md`; (2) check which packages import or depend on the domain area described in `idea.md`; (3) if ambiguous, list the candidate packages as an Open Question for the human reviewer rather than guessing."* The last clause is critical — it turns an ambiguous situation into an explicit decision point rather than a silent arbitrary choice.

#### A4. The Complexity Gate Concession Lacks Teeth

**Defender's proposed fix for Issue 12:**
> *"If `idea.md` describes more than 3–4 major components, note this in the Executive Summary. Prioritize research depth on the components with highest risk or uncertainty."*

"Prioritize research depth" is vague — depth of what? More web searches? More files read? Longer feasibility notes? And "highest risk or uncertainty" presupposes the agent can assess risk **before** doing the research, which is circular. You need the research to know the risk, but you need the risk assessment to know where to research deeply.

The defender's argument against a hard cap ("would silently ignore potentially critical components") is valid, but their alternative is toothless. The agent reads the instruction, generates research on everything anyway (because it can't pre-assess risk), and adds a note saying "this was a broad idea" — which the human already knew from writing it.

**Better fix:** Make the gate procedural, not advisory: *"If `idea.md` describes more than 4 major components, first complete Steps 1–2 at surface level for all components. Then use the initial findings to rank components by uncertainty (unknowns or conflicting signals) and complete Steps 3–5 in depth for only the top 3. In the Executive Summary, list all components with their coverage level (deep/surface) and recommend follow-up research passes for surface-level components."* This is a two-pass approach that resolves the circularity.

### B. New Vulnerabilities Introduced by Proposed Changes

#### B1. The "Conflicts & Tradeoffs" Section Creates an Unresolvable Mandate

The defender's most confident contribution — the new "Conflicts & Tradeoffs" subsection — asks the agent to *"state which you recommend and why."*

This is a category error. The researcher agent is explicitly positioned as an evidence-gatherer: *"Your job is to gather evidence... so that downstream agents can make informed decisions without guessing."* Making recommendations is the product-manager's and architect's job. If the researcher starts recommending resolutions to conflicts, it's overstepping its role and potentially anchoring downstream agents on premature conclusions.

**Failure scenario:** Web research finds both REST and GraphQL as viable API patterns. The codebase uses REST. The researcher "recommends" REST because of codebase alignment. The architect reads the recommendation and doesn't seriously evaluate GraphQL, even though for this specific feature GraphQL would have been significantly better. The researcher's recommendation — based on pattern matching, not architectural analysis — becomes a de facto decision that no one explicitly made.

**Better approach:** Document the conflict without recommending. The subsection should say: *"When findings from different sources conflict, document both positions with their supporting evidence, and flag the conflict as an explicit decision point in the Open Questions section. Do not recommend a resolution — that is the architect's role."*

#### B2. The Feasibility "Assumes" Note Adds Noise Without Accountability

The defender's fix for Issue 5 adds: *"For each component rated L or above, include a brief 'assumes:' note."*

But only for L or above? A component rated "S" with a wrong assumption is more dangerous than an "L" with an explicit one, because nobody scrutinizes "S" items. The threshold creates a false sense that small estimates are assumption-free.

Moreover, "a single developer familiar with the project's primary stack" is itself an assumption that may not hold. If the project requires integrating an unfamiliar API or learning a new framework, the "familiar with the stack" anchor is actively misleading.

**Better approach:** Drop the calibration persona entirely. Instead: *"For each component, state the primary assumption driving the estimate. If the estimate depends on team familiarity with a specific technology, say so."* Apply it uniformly. Let the human recalibrate based on their actual context.

### C. Insufficient Rebuttals

#### C1. Issue 2 (Output Validation) — "The Template IS Validation" Is Empirically False

The defender claims: *"LLMs generating from templates don't silently drop sections."*

This is demonstrably wrong. LLMs routinely drop, merge, or truncate sections when:
- The context window is near capacity (common after extensive codebase exploration in Step 2)
- A tool call returns an unexpectedly large result that pushes the template out of attention
- The model is a smaller/cheaper variant (the spec says `model: inherit`, so ANY model could run this)

The defender's argument that "the human review gate catches errors" is a last line of defense, not an excuse to skip intermediate validation. The human reviewer has to **know** something is missing to catch it. If the Quality Commands table is simply absent from `research.md`, a reviewer skimming for the Executive Summary and Feasibility Assessment may not notice. The human gate is a review of research quality, not a structural completeness audit.

The fix doesn't need to be complex. One line in the Output section: *"Before finalizing, verify that research.md contains all six required sections: Executive Summary, External Research, Codebase Analysis, Quality Commands, Feasibility Assessment, Open Questions. If any section is empty, add an explicit 'Section intentionally empty: [reason]' note."* This is cheap, takes zero extra tool calls, and makes gaps visible to human reviewers.

---

#### C2. Issue 6 (Crash Recovery) — "Orchestrator's Responsibility" Ignores the Spec's Own State Management

The defender argues crash recovery belongs in the orchestrator, not the agent. But the spec **already puts state management in the agent** — the `.spec-drive-state.json` update and the `.progress.md` append are agent-side state operations. The spec can't simultaneously say "the agent manages state" and "crash recovery isn't the agent's concern."

The defender's implicit-crash-detection argument — "if `research.md` exists but `awaitingApproval` is not set, the orchestrator knows" — actually **doesn't work** as described. If the agent crashes after writing a complete `research.md` but before setting `awaitingApproval`, the orchestrator sees: file exists, no approval flag. Is that a crash? Or did the agent intentionally skip approval (a bug)? Or is the agent still running? The orchestrator can't distinguish these states.

I'm not asking for full per-step checkpointing. But the spec should acknowledge the failure mode: *"If re-invoked after a partial run, check whether `research.md` already exists. If it does and `awaitingApproval` is not set, the previous run likely failed mid-execution. Regenerate `research.md` from scratch rather than assuming the existing file is complete."* This is one sentence. It costs nothing. It prevents the specific scenario where a downstream agent consumes a truncated `research.md` because nobody realized the previous run crashed.

#### C3. Issue 11 (Summary Ordering) — The Defender's Confidence Is Unwarranted

The defender states: *"LLMs routinely generate documents with sections assembled out of sequential order."*

This is true for models generating complete documents in a single output. But this agent **writes to a file using a tool call.** The agent must either (a) compose the entire document in one `Write` call, or (b) write incrementally section by section. 

In case (a), the model generates the full text in context before calling `Write` — the defender is correct that summary-last generation works. But for long research documents, this means the **entire document** sits in the generation buffer before being written, increasing the risk of truncation in the model's output.

In case (b), the agent writes sections as it completes them, which is more robust for long documents. But then the Executive Summary must either be written first (before analysis, the critique's concern) or require an edit/prepend operation after all other sections are written (adding tool-call complexity).

The spec gives no guidance on which approach to use, which means the model will default to whatever is easiest — typically (a) for short research and (b) for long research. The risk is specifically with long research documents where approach (b) kicks in.

**Minimal fix:** Add to Step 6: *"Compose the Executive Summary after completing Steps 1–5. Write the full research.md in a single operation."* This pins the behavior to approach (a) and eliminates the ambiguity without moving the summary to the end of the template.

## New Issue the Defender Introduced

### D1. The Defender's "Conflicts & Tradeoffs" Section Breaks the Template Contract

The defender proposes adding a new section "between Codebase Analysis and Quality Commands" but doesn't update the output template in the spec. If this section is added to the instructions but not to the template, the agent must decide whether to include it. Some runs will include it, some won't. Downstream agents parsing `research.md` programmatically (a real use case in a spec-driven pipeline) will encounter inconsistent structure.

**Fix:** If the section is added, it must appear in both the execution steps AND the output template. And it should be listed in any structural completeness check.
