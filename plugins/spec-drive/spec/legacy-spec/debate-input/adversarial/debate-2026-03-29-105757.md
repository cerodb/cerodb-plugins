## Round 1 — CRITIC

# Critique: Product Manager Agent Specification

## Content Type

This is a **process specification / agent prompt** that defines how an AI agent should transform two input files (`idea.md`, `research.md`) into a structured `requirements.md`. "Improved" means: the spec produces correct, complete, unambiguous requirements reliably across diverse inputs, with no silent failure modes.

---

## Critical Issues (ordered by damage potential)

### 1. The "Sufficiency Check" Is Undefined and Will Either Block Everything or Block Nothing

**Section:** "Before proceeding, do a sufficiency check"

The spec says to stop if input is "too thin to ground real requirements" or if "a critical section is absent or empty." But it never defines:
- What constitutes "too thin"
- Which sections are "critical"
- What threshold separates "proceed with caveats" from "stop entirely"

**Failure scenario:** Agent A receives an `idea.md` with a 3-line vision and no constraints section. It stops. Agent B receives the same file and proceeds, because its interpretation of "too thin" differs. The spec produces inconsistent behavior across runs and models. Worse — a weaker model will almost always proceed (bias toward generating output), making the gate useless precisely when it's most needed.

**Fix:** Define explicit structural requirements. Example: "`idea.md` must contain a non-empty `## Vision` and `## Constraints` section. `research.md` must contain a non-empty `## Executive Summary` and `## Feasibility Assessment`. If any of these are missing or under 50 words, stop and report."

---

### 2. No Conflict Resolution Strategy — Just "Surface It"

**Section:** Step 6, "Unresolved Conflicts / Clarifications Needed"

The spec tells the agent to surface conflicts between `idea.md` and `research.md` but gives zero guidance on what to do with the *requirements that depend on the conflicted information*. Should the agent:
- Write the requirement anyway with a caveat?
- Omit the requirement entirely?
- Write two competing requirements?

**Failure scenario:** `idea.md` says "must support SQLite" and `research.md` says "SQLite cannot handle the expected concurrent load." The agent writes FR-3 requiring SQLite, notes the conflict in Section 7, and the downstream architect implements SQLite because the FR table is what they read. The conflict note is buried and ignored.

**Fix:** Require that any FR or NFR derived from conflicted sources must be marked `[BLOCKED — see Unresolved Conflicts #N]` in the FR table itself, not just cross-referenced in a separate section. Blocked FRs cannot be prioritized as `High`.

---

### 3. The 5-7 AC Split Rule Has No Enforcement and Creates Numbering Chaos

**Section:** Step 1 — "If one story would need more than 5-7 acceptance criteria, split it into smaller stories"

This is advisory, not structural. There's no mechanism to detect when the threshold is crossed before the ACs are written. The agent will write 12 ACs, realize it should have split, and then either:
- Ignores the rule (most likely)
- Retroactively splits, causing AC renumbering that breaks FR `Source` references already drafted

**Failure scenario:** US-3 gets split into US-3a and US-3b mid-generation. FR-7 references `US-3` which no longer exists. The self-check at the end doesn't validate FR→US reference integrity — it only checks that "every FR has a Source," not that the source ID actually exists.

**Fix:** Add to the self-check: "every FR `Source` value must reference a valid US-N identifier defined in the User Stories section." Also, prescribe a two-pass approach: draft feature areas and story boundaries first, then write ACs — don't split retroactively.

---

### 4. The Self-Check Is Incomplete — It Misses the Most Common Corruption Modes

**Section:** "Self-Check"

The self-check verifies:
- Every US has ACs ✓
- Every FR has a Source ✓
- Every NFR has a target ✓
- Every Success Criterion references FR/NFR IDs ✓
- Unresolved ambiguities are surfaced ✓

It does **not** verify:
- FR `Source` IDs actually exist (phantom references)
- AC numbering matches its parent story (AC-3.1 actually lives under US-3)
- No duplicate FR/NFR IDs
- Success Criteria FR/NFR references actually exist
- Dependencies section is non-empty (every project has at least one)
- Out of Scope section is non-empty
- Word count is under 3000

**Failure scenario:** The agent writes `AC-2.1` under `US-3` (copy-paste drift, extremely common in LLM output). The downstream QA engineer uses AC IDs to build test cases and assigns AC-2.1 to the wrong feature. Nobody catches it because the self-check doesn't validate AC-parent alignment.

**Fix:** Add these checks explicitly. The self-check is the agent's last line of defense — it must be exhaustive for structural integrity, not just spot-check a few fields.

---

### 5. Priority Rubric Will Produce All-High Documents

**Section:** Step 4 — Priority rubric

The rubric says `High` = "without it, the project cannot deliver its core value." For most feature specs, 60-80% of requirements will meet this bar because `idea.md` typically describes what's core to the project. The spec acknowledges this: "If more than half of FRs look High, note that the scope may be too broad."

But "note it" is toothless. The agent notes it, continues, and the architect receives a document where everything is High priority with a footnote saying "scope might be too broad." That footnote changes nothing downstream.

**Failure scenario:** 8 of 10 FRs are High. The task planner treats them all as mandatory for MVP. The project takes 3x longer than expected because there was no real prioritization — just a label that said "everything matters."

**Fix:** Enforce a hard cap: no more than 40% of FRs can be `High`. If exceeded, the agent must either downgrade or explicitly split into "MVP" and "Post-MVP" groupings with rationale. Force the prioritization decision rather than deferring it.

---

### 6. "Do NOT Include Implementation Details" Is Unenforceable and Contradicts NFR Requirements

**Section:** Constraints — "requirements describe WHAT, not HOW"

The NFR table requires specific metrics and targets (e.g., "response time < 200ms"). But what's the response time *of*? The API endpoint? The CLI command? The database query? Specifying this requires referencing implementation architecture, which the spec forbids.

**Failure scenario:** NFR-2 says "Response time < 200ms." The architect interprets this as end-to-end API latency. The original intent was CLI command execution time. The requirement was technically pure ("what, not how") but ambiguous precisely because it lacked the implementation context needed to interpret it.

**Fix:** Allow NFRs to reference *system boundaries* (e.g., "from user input to visible output") without prescribing implementation. Add guidance: "NFRs must specify the measurement boundary — what is being timed, from where to where."

---

### 7. No Versioning or Idempotency — Overwrites Destroy Previous Work

**Section:** Output — "Write `{basePath}/requirements.md`"

The spec says "write" — not "create if absent" or "update." If the agent is re-run (common in iterative workflows), it silently overwrites the previous `requirements.md` with no diff, no backup, no changelog.

Meanwhile, `.progress.md` says "append only, never delete existing learnings." So the progress file accumulates history but the actual deliverable doesn't.

**Failure scenario:** A human reviews `requirements.md`, adds comments or annotations inline, then re-runs the agent after updating `research.md`. All human annotations are destroyed. No record of what changed between versions.

**Fix:** Either (a) require the agent to read existing `requirements.md` first and produce a diff/changelog section, or (b) write to `requirements-v{N}.md` with incrementing versions, or (c) at minimum, mandate a backup before overwrite.

---

### 8. The 3000-Word Limit Is Arbitrary and Has No Degradation Strategy

**Section:** Constraints — "Keep the document under 3000 words"

For a complex project with 8+ feature areas, 3000 words forces aggressive compression. But the spec gives no guidance on *what to cut*. Does the agent drop NFRs? Shorten AC descriptions? Remove the glossary?

**Failure scenario:** The agent hits 2900 words at Step 5 (NFRs). It skips the glossary, writes a skeletal Out of Scope section, and omits Dependencies entirely. The downstream architect has no dependency list and misses a critical external API requirement.

**Fix:** Define a priority order for sections when space is constrained: User Stories + ACs → FRs → Dependencies → NFRs → Out of Scope → Success Criteria → Glossary. Or better: make the limit advisory ("target 3000 words; if exceeded, flag it but do not truncate critical sections").

---

### 9. Role Derivation Is Underspecified

**Section:** Step 2 — "Derive roles from the inputs when possible. If a role is implied but not explicit, define it in the Glossary."

This creates a silent failure mode: the agent invents a role name (e.g., "Platform Operator"), defines it in the glossary, and now downstream agents treat it as a real stakeholder. There's no validation that the role maps to an actual user of the system.

**Failure scenario:** The agent infers a "System Administrator" role from a constraint about config files. It writes 3 user stories for this role. The actual project has no admin — it's a single-user CLI tool. Three user stories and their FRs are waste.

**Fix:** Require that any role not explicitly named in `idea.md` must be flagged as `[Inferred — verify with stakeholder]` in both the user story and the glossary.

---

### 10. `.progress.md` Creation Is a Side Effect With No Error Handling

**Section:** Progress Update

"If it does not exist yet, create it first." But there's no specified structure for the initial creation. What sections does a new `.progress.md` contain? What if the file exists but is malformed? What if the write fails (permissions, disk)?

**Failure scenario:** Two agents race — the PM agent creates `.progress.md` while the research agent is also trying to write to it. The file ends up with interleaved content. Neither agent checks for concurrent access.

**Fix:** Define the canonical initial structure of `.progress.md`. Add a note that this file should be treated as append-only with section headers that include timestamps, making concurrent writes at least detectable if not safe.

## Round 1 — DEFENDER

# Defense: Product Manager Agent Specification

## The Critic's Blindspot — What They Missed

Before addressing the ten points: **the critic focused almost entirely on structural integrity and mechanical failure modes while ignoring the most consequential gap in this specification — there is no guidance on requirements granularity.**

The spec tells the agent to derive Functional Requirements from User Stories, but never defines the expected granularity of an FR. A user story like *"As a developer, I want to deploy the application so that users can access it"* could yield:
- **1 FR**: "System must be deployable to production" (too coarse for task planning)
- **12 FRs**: Docker support, CI/CD pipeline, env variable management, health checks, logging config, TLS termination, rollback mechanism, etc. (too granular, crosses into implementation)

This variance matters more than any of the critic's points because **every downstream agent — architect, task planner, executor — calibrates its output to the FR count and granularity.** If the PM agent produces 6 FRs in one run and 25 in another for equivalent inputs, the entire spec-driven pipeline becomes unpredictable. The critic obsessed over AC numbering alignment (Point 3) and phantom ID references (Point 4) — real but catchable errors — while missing that the *semantic density* of requirements is uncontrolled.

**Fix:** Add granularity guidance: *"Each FR should describe a single testable capability at the level where a developer could estimate it in hours to days, not minutes or weeks. If an FR takes less than an hour to verify, it's too granular — merge up. If it would take more than a week to verify, it's too coarse — decompose."*

This matters more than every structural check combined because structural errors produce documents that *look wrong* (and get caught), while granularity drift produces documents that *look fine* but derail downstream work silently.

## Point-by-Point Response

### Point 1: Sufficiency Check — PARTIAL CONCEDE, REJECT THE FIX

The critic correctly identifies that "too thin" is subjective. I concede that the sufficiency check needs more structure.

However, the proposed fix — mandating specific section headers (`## Vision`, `## Constraints`) and a 50-word minimum — is **brittle and counterproductive**. This spec is designed to work across diverse project types. A well-written `idea.md` for a simple CLI tool might use `## Goal` instead of `## Vision`, or express its entire constraint set in 40 precise words. Rejecting that file because it uses different headers or is 10 words short of an arbitrary minimum creates false negatives.

**Better fix:** Define sufficiency functionally rather than structurally:

> `idea.md` must contain: (a) a stated goal or vision for what the project achieves, (b) at least one explicit constraint or boundary. `research.md` must contain: (a) findings about feasibility or existing solutions, (b) identification of at least one risk or open question. If any of these four elements cannot be found regardless of heading structure, stop and report which element is missing.

This checks for *content* rather than *formatting*, which is what actually matters for grounding requirements.

### Point 2: Conflicts Noted But Not Blocked — CONCEDE

The critic nails this. The failure scenario is specific (SQLite capacity conflict), the affected party is clear (downstream architect implements the wrong thing), and the likelihood is high (architects read the FR table, not the appendix).

I accept the critic's proposed fix with one refinement: rather than just `[BLOCKED]`, use `[CONFLICTED — see Unresolved #N]` because "blocked" implies it cannot proceed at all, whereas some conflicts may have a likely resolution once a stakeholder weighs in. The FR should still appear in the table (so nothing is invisible) but should be **visually flagged and excluded from priority sorting** until resolved.

**Specific addition to Step 4:**

> If an FR depends on information where `idea.md` and `research.md` conflict, mark it `[CONFLICTED — see Unresolved #N]` in the Requirement column. Conflicted FRs must have Priority set to `Blocked` and cannot be counted toward scope assessment until resolved.

### Point 3: Split Rule and Numbering Chaos — REJECT THE PROBLEM, ACCEPT THE SELF-CHECK ADDITION

The critic's scenario — FR-7 referencing a now-nonexistent US-3 after a retroactive split — misunderstands the spec's step ordering. The steps are sequential:

1. **Step 1**: Identify feature areas and story boundaries (including splits)
2. **Step 2**: Write user stories with final numbering
3. **Step 3**: Write ACs
4. **Step 4**: Build FR table with Source references

The FR table is written *after* stories are finalized. The agent doesn't draft FR references and then retroactively renumber — it establishes story boundaries in Step 1, writes them in Step 2, and only then builds the FR table. The "chaos" scenario requires the agent to violate the prescribed step order.

That said, the critic's suggestion to add **FR→US reference validation** to the self-check is genuinely useful — not because the split rule causes chaos, but because LLMs do occasionally produce phantom references through generation drift. This is cheap insurance.

**Addition to self-check:** *"Every FR `Source` value must reference a US-N identifier that exists in the User Stories section."*

I reject the "two-pass approach" suggestion because that's already what the step ordering provides.

### Point 4: Self-Check Is Incomplete — PARTIAL CONCEDE

I concede that the self-check should include:
- **FR Source ID validation** (phantom references) — yes, this catches real LLM drift
- **AC-parent alignment** (AC-X.Y lives under US-X) — yes, this is a common LLM copy-paste error
- **No duplicate FR/NFR IDs** — yes, trivial to check and catches real errors
- **Success Criteria reference validation** — yes, same logic as FR Source validation

I **reject** the remaining suggestions:
- **Dependencies section non-empty**: Not every project has external dependencies. A self-contained CLI tool with no API calls or external libraries has none. Forcing a non-empty check produces fabricated dependencies.
- **Out of Scope non-empty**: Same argument. For a tightly-scoped spec, the boundaries might be implicit from idea.md. The section can legitimately be "No explicit exclusions beyond what idea.md constrains."
- **Word count under 3000**: Turning a constraint into a self-check item converts it from a writing guideline into a hard gate. If the document is 3,100 words and structurally sound, failing the self-check and... what? Truncating? That's Point 8's problem, not a self-check problem.

The critic asked for "exhaustive structural integrity" checks. I agree on structural (ID references, numbering), disagree on content-presence (sections must be non-empty). Structural corruption is silent; empty sections are visible.

### Point 5: Priority Rubric Produces All-High — REJECT

The critic's proposed fix — a 40% hard cap on High-priority FRs — is **actively harmful** and I'll defend this strongly.

If a project genuinely has 8 critical requirements out of 10, artificially downgrading 3 of them to Medium to satisfy a quota **lies to the downstream architect**. The architect deprioritizes FR-6 ("authentication") because the PM agent was forced to label it Medium to meet a cap. The delivered system has no auth. The cap created a worse outcome than the all-High document.

The critic's failure scenario ("task planner treats everything as mandatory for MVP, project takes 3x longer") is a **planning failure, not a requirements failure**. If everything genuinely is High, the correct response is to reduce scope (fewer features), not to misrepresent priority.

The spec *already* handles this: "If more than half of FRs look High, note that the scope may be too broad for one spec." The critic calls this "toothless," but I disagree — it signals to the human stakeholder that the project needs scoping conversation. That's the PM agent's job: surface reality, not force artificial constraints.

**What I will strengthen:** Change "note that the scope may be too broad" to a more actionable output:

> If more than 50% of FRs are High, add a `## Scope Advisory` section that: (a) explicitly lists which FRs could be deferred to a follow-up spec without breaking the core value proposition, and (b) proposes a recommended MVP boundary. This section is advisory — the stakeholder decides, but the agent must make the trade-off visible.

This forces the prioritization *reasoning* without forcing incorrect priority labels.

### Point 6: NFRs Need Measurement Boundaries — CONCEDE

The critic is right. "Response time < 200ms" is meaningless without knowing what's being measured from where to where. And the "no implementation details" constraint does create tension with NFRs that need measurement context.

I accept the fix. Specifying measurement boundaries ("from user input to visible output," "from API request received to response sent") is *context*, not implementation. It answers "what are we measuring?" without answering "how do we build it?"

**Addition to Step 5 rules:**

> Every NFR with a performance, latency, or throughput target must specify the measurement boundary: what is being measured, from which system entry point to which exit point. Measurement boundaries describe observable behavior, not internal architecture.

### Point 7: Overwrite Destroys Previous Work — REJECT

The critic's scenario — a human adds inline annotations to `requirements.md`, re-runs the agent, and loses everything — misunderstands the operational context of this spec.

This agent operates within a **spec-driven development pipeline** where the spec directory is under version control (git). Every overwrite is recoverable via `git diff` or `git checkout`. The standard workflow is:

1. Agent generates `requirements.md`
2. Human reviews (via git diff, PR comments, or separate review notes)
3. If inputs change, agent regenerates
4. Git shows exactly what changed

Human annotations belong in **review comments, PR discussions, or a separate `review-notes.md`** — not inline in a generated artifact. Encouraging inline edits to generated files is an anti-pattern because it creates merge conflicts between human and machine output.

The `.progress.md` append-only rule exists precisely because that file *accumulates* cross-run context. `requirements.md` is a *derived artifact* — it reflects the current state of its inputs, not its own history.

I reject all three proposed fixes:
- (a) Reading existing `requirements.md` and producing a diff adds complexity for marginal value when git already provides this
- (b) `requirements-v{N}.md` creates file proliferation that confuses downstream agents about which version is current
- (c) Mandatory backup before overwrite duplicates what git does

**What I will add** (because the critic's instinct about traceability is sound even if the fix is wrong):

> Add a `generated` timestamp to the frontmatter. This, combined with version control, provides full history.

The frontmatter already has `created` — I'd rename it to `generated` to clarify this is a regenerated artifact, not a one-time creation.

---

### Point 8: 3000-Word Limit Has No Degradation Strategy — PARTIAL CONCEDE

The critic is right that the spec gives no guidance on what to cut when space is tight. However, making the limit "advisory" (as the critic suggests) is equivalent to removing it — advisory limits are ignored limits, especially by LLMs biased toward verbosity.

I concede that a **section priority order** is needed for graceful degradation. The critic's proposed order is close but wrong — Dependencies should not outrank NFRs. A missing dependency is more damaging than a missing performance target because dependencies represent hard blockers.

**Fix:** Add degradation guidance:

> If approaching the 3000-word limit, compress in this order (last = cut first): Glossary → Out of Scope → Success Criteria → NFRs → Dependencies → FR table → User Stories + ACs. User Stories with ACs are never truncated; they are the core deliverable. If the document cannot fit within 3000 words even after compressing lower-priority sections, note the overflow and include all content — completeness overrides brevity.

That last sentence converts the hard limit to a soft limit with escalation, which is the right behavior.

### Point 9: Inferred Roles — CONCEDE (MINOR)

The critic's scenario is plausible (inventing a System Administrator for a single-user tool) and the fix is cheap and low-risk. I accept the proposed fix.

**Addition to Step 2:**

> Any role not explicitly named in `idea.md` or `research.md` must be flagged as `[Inferred]` in the user story and defined with a `[Verify with stakeholder]` note in the Glossary.

This is a minor issue — the critic correctly rated it Medium — but the fix costs nothing and prevents real waste.

### Point 10: `.progress.md` Has No Structure — REJECT

The concurrent access scenario (PM agent and research agent racing to write `.progress.md`) is **not realistic in this pipeline**. The spec-driven workflow is sequential: research runs first, produces `research.md`, then the PM agent runs. They do not operate on the same spec simultaneously. The `basePath` is a single spec directory with a defined phase progression.

The "malformed file" concern is similarly low-impact — `.progress.md` is append-only metadata, not a critical artifact. If it's garbled, downstream agents read `requirements.md`, not `.progress.md`.

That said, the critic is right that the *initial structure* is undefined. This is worth one sentence:

**Addition to Progress Update:**

> If `.progress.md` does not exist, create it with a `# Progress` heading and a `## Learnings` subsection. Then append your entry.

I reject the concurrency handling because it solves a problem that doesn't exist in the pipeline's execution model.

## Attacking the Defense: Where the Fixes Fail and the Rebuttals Don't Hold

---

### A. Weak Fixes for Conceded Points

#### A1. The "Functional Sufficiency Check" (Point 1 Fix) Is Just as Subjective — With More Words

The defender replaced "too thin" with four functional criteria: a stated goal, at least one constraint, feasibility findings, and at least one risk or open question. Sounds better. It isn't.

"Findings about feasibility or existing solutions" — what counts as a finding? A single sentence saying "No existing tools were found" is a finding. It tells you nothing useful, but it passes the gate. The defender's fix replaces vague *adjectives* ("too thin") with vague *nouns* ("findings," "identification"). The structure changed; the ambiguity didn't.

**Failure scenario:** `research.md` contains: "## Feasibility: This appears feasible based on initial review. ## Risks: The timeline is tight." Two sentences. Both elements are "present." The gate passes. The PM agent now writes requirements grounded in a research document that says essentially nothing. Every FR inherits the unearned confidence of a gate that checked for *existence* rather than *substance*.

**What actually works:** The gate needs to test for *groundability*, not presence. Add: "For each element, the agent must be able to cite a specific claim from the input when writing at least one FR or NFR. If the agent cannot trace any requirement back to a concrete input claim, the input is insufficient." This makes the test functional at the *output* stage rather than trying to evaluate input quality in the abstract — which LLMs are bad at.

---

#### A2. The Conflicted-FR Fix (Point 2) Introduces a New Priority Value With No Downstream Contract

The defender proposed adding `Priority: Blocked` for conflicted FRs. The existing priority rubric defines three values: High, Medium, Low. Now there's a fourth — `Blocked` — that appears nowhere in the priority rubric definition, has no defined semantics for downstream consumers, and breaks any tooling or agent that parses the FR table expecting a three-value enum.

**Failure scenario:** The architect agent reads the FR table and filters for `High` priority items to design first. FR-4 (authentication, genuinely critical) is marked `Blocked` because there's a conflict about *which* auth method to use — not whether auth is needed. The architect skips it entirely. Auth isn't designed. The task planner never sees it. The conflict was about a detail; the consequence was omission of an entire capability.

**Fix the fix:** Don't overload the Priority column. Keep Priority as the three-value assessment of importance. Add a separate `Status` column with values `Ready` or `Blocked`. A requirement can be `High` priority *and* `Blocked` — that combination communicates urgency ("this is critical AND we can't proceed until the conflict is resolved"), which is far more useful than collapsing both dimensions into one field.

---

#### A3. The Scope Advisory (Point 5 Fix) Asks the PM Agent to Do Scope Negotiation — A Different Job

The defender replaced the toothless "note it" with a `## Scope Advisory` section that "proposes a recommended MVP boundary" and "explicitly lists which FRs could be deferred."

This is scope negotiation. It requires understanding business strategy, customer segmentation, go-to-market timing, and competitive dynamics. The PM agent's input is two files — `idea.md` and `research.md` — neither of which necessarily contains this context. The agent will *guess* what's deferrable based on its general training, not on actual business knowledge.

**Failure scenario:** The agent defers FR-5 (export functionality) to post-MVP because it seems like a "nice to have." But `idea.md` mentions a key customer who specifically requires export for compliance. The agent didn't connect those dots because the compliance requirement was in the Constraints section, not explicitly linked to the export feature area. The Scope Advisory now actively misleads the stakeholder by presenting a plausible-looking but wrong MVP boundary.

**The defender's own logic defeats them here.** They argued in Point 7 that "human annotations belong in review comments, not inline in generated artifacts." By the same logic, MVP boundary decisions belong in stakeholder conversations, not in a generated artifact. The Scope Advisory should *frame the question* (present the options and trade-offs) without *recommending an answer*. Replace "proposes a recommended MVP boundary" with "presents candidate deferral options with stated trade-offs for stakeholder decision."

---

#### A4. The Degradation Priority Order (Point 8 Fix) Puts Glossary Last — It Should Be First to Survive

The defender's compression order (cut first → cut last): Glossary → Out of Scope → Success Criteria → NFRs → Dependencies → FR table → User Stories + ACs.

The Glossary is the first casualty. But **the Glossary is the only section that prevents misinterpretation of every other section.** If the spec uses domain terms (which any non-trivial project will), and the Glossary is cut, then the FR table, NFRs, and user stories all become ambiguous to the downstream architect who doesn't share the PM agent's implicit definitions.

**Failure scenario:** The spec uses "tenant" throughout — 4 user stories reference tenant isolation, 3 FRs specify tenant-scoped data access. The Glossary defined "tenant" as "a single organization account, not an individual user." The Glossary gets cut for space. The architect implements per-user isolation instead of per-organization isolation. Every FR technically passes its AC but the system is architecturally wrong.

**Fix:** Move Glossary to survive before NFRs. Cut order should be: Out of Scope → Success Criteria → NFRs → Glossary → Dependencies → FR table → User Stories + ACs. Out of Scope is the most safely cut — it says what the project *isn't*, which is useful but not load-bearing for implementation.

---

### B. New Vulnerabilities Introduced by Defender's Proposals

#### B1. The Granularity Guidance ("Hours to Days") Creates a Temporal Estimation Requirement the Agent Can't Fulfill

The defender's own new issue — requirements granularity — proposed: "Each FR should describe a single testable capability at the level where a developer could estimate it in hours to days, not minutes or weeks."

This requires the PM agent to *estimate implementation effort* for each FR. But the PM agent is explicitly told "do NOT include implementation details — requirements describe WHAT, not HOW." Estimating that something takes "hours vs. weeks" requires knowing *how* it would be built. A requirement like "System must authenticate users via OAuth 2.0" might take 2 hours with an existing library or 2 weeks if building from scratch. The PM agent has no way to judge this without implementation context it's forbidden from considering.

**Failure scenario:** The agent writes FR-3: "System must support real-time data synchronization." Is that hours or weeks? Depends entirely on the architecture. The agent, following the granularity rule, either (a) breaks it into micro-requirements that prescribe implementation ("must use WebSocket connections," "must maintain event queue") — violating the no-implementation constraint, or (b) leaves it coarse and adds a note saying "granularity uncertain" — making the guidance useless.

**Fix:** Define granularity in terms of *verification scope*, not *implementation effort*: "Each FR should be verifiable with a single test scenario. If verifying an FR requires testing multiple independent behaviors, decompose it. If two FRs would always be tested together and cannot independently fail, merge them." This keeps granularity in the requirements domain (testability) rather than requiring implementation knowledge (effort estimation).

---

#### B2. The `generated` Timestamp (Point 7 Fix) Gives False Confidence About Traceability

The defender renamed `created` → `generated` and claims this "combined with version control, provides full history." It does not. A timestamp tells you *when* the file was last written. Git tells you *that* it changed. Neither tells you *why* it changed — which inputs were different, which requirements were added/removed/modified, and whether those changes were intentional.

More importantly, the defender's entire Point 7 rebuttal assumes git is in use. The spec says the `basePath` is a directory — it never mandates version control. The cross-CLI portability section says another agent should be able to continue "without this conversation" — it does not say "with access to the git history of this directory."

**Failure scenario:** Team uses the spec pipeline on a shared drive or in a CI/CD temp directory. No git. Agent regenerates `requirements.md` after `research.md` is updated. Previous requirements vanish with no record. The `generated` timestamp says "March 29, 2026" — that's all anyone knows.

**The defender argued this isn't a problem because the pipeline uses git.** But the spec is explicitly designed for cross-CLI portability. Portability means not assuming the execution environment. If traceability requires git, the spec should say so in Dependencies, not assume it silently.

**Fix:** Either (a) add "version control on the spec directory" as an explicit dependency in the spec, or (b) have the agent read any existing `requirements.md` before overwriting and note material changes in `.progress.md` (which is already append-only). Option (b) is cheap and self-contained.

### C. Insufficient Rebuttals

#### C1. Point 3 Rebuttal Relies on Step Ordering That LLMs Don't Reliably Follow

The defender argued that the split/renumber problem can't happen because "the steps are sequential" — Step 1 sets boundaries, Step 2 writes stories, Step 4 builds the FR table. Therefore, FRs are always written after story IDs are finalized.

This assumes the agent follows the steps in strict sequence with no backtracking. LLMs do not work this way. They generate text in a stream. When an LLM is in Step 4 writing FR-7 and references `US-3`, it may realize that US-3 should have been split. It doesn't go back and re-execute Steps 1-3 — it either continues with the inconsistency or attempts an inline correction that creates the exact chaos the critic described.

The defender accepted the self-check addition (FR→US reference validation) which partially mitigates this. But the self-check is a *detection* mechanism, not a *prevention* mechanism. The agent detects the broken reference and then... what? The spec has no error recovery protocol. Does it re-run from Step 1? Edit the FR table? Add a note? The self-check catches the problem but the spec provides no remediation path.

**What's still needed:** Add an explicit recovery instruction: "If the self-check detects an inconsistency (phantom reference, misaligned AC), fix the inconsistency by adjusting the referencing element (FR Source, AC number) to match the source of truth (User Stories), then re-run the self-check. Do not adjust User Story IDs to match downstream references."

---

#### C2. Point 7 Rebuttal — "Human Annotations Are an Anti-Pattern" Doesn't Match Reality

The defender argued that humans shouldn't annotate `requirements.md` inline because "human annotations belong in review comments, PR discussions, or a separate `review-notes.md`." This is an idealized workflow that ignores how people actually work with documents.

The spec targets cross-CLI portability. Some of those CLIs won't have PR review systems. Some users will open the file in their editor, add a note ("check with legal on this"), and save it. The defender is prescribing a workflow discipline that the spec has no authority to enforce and no mechanism to detect violations of.

More critically, the defender's own spec writes to `.progress.md` with "append only, never delete" semantics — acknowledging that some files need history preservation. If `.progress.md` deserves append-only protection, `requirements.md` — the actual deliverable — deserves at least a backup-before-overwrite. The defender's position is internally inconsistent: generated metadata gets history protection but the generated artifact doesn't.

**The concession needed here isn't versioned filenames or git dependency.** It's minimal: before writing `requirements.md`, if the file already exists, log a one-line entry in `.progress.md`: "Previous requirements.md overwritten (generated: [old timestamp])." This costs nothing, requires no git, and creates a minimal audit trail.

---

#### C3. Point 10 Rebuttal — "Pipeline Is Sequential" Assumes a Deployment Model the Spec Doesn't Mandate

The defender rejected the concurrency concern because "the spec-driven workflow is sequential: research runs first, then PM agent runs." But this operational assumption appears *nowhere in the spec itself*. The spec defines its inputs (`idea.md`, `research.md`) and its output (`requirements.md`, `.progress.md`). It does not define its place in a pipeline or assert that no other agent operates on the same `basePath` concurrently.

The defender is defending the spec based on knowledge of the *system* it operates in, not based on what the *spec actually says*. This is exactly the kind of hidden context the spec's own Cross-CLI Portability section warns against: "no references to hidden context."

This doesn't require a full concurrency model. It requires one sentence: "This agent assumes exclusive write access to `{basePath}/requirements.md` and `{basePath}/.progress.md` during execution." That's a stated assumption, not a solution — and stated assumptions are exactly what good specs contain.
